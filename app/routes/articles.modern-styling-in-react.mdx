## The Problem with Traditional E-Commerce Search

Traditional e-commerce search relies heavily on keyword matching and basic filters. When a user searches for "comfortable running shoes for flat feet," a keyword-based system struggles to understand the semantic intent. It might return results containing the exact words but miss highly relevant products that use different terminology.

This is where Retrieval-Augmented Generation (RAG) systems shine. By combining the retrieval capabilities of vector databases with the reasoning power of Large Language Models, we can build search systems that truly understand what users are looking for.

## Architecture Overview

When building [Nexora](https://nexora-api-73508499658.us-central1.run.app/), our AI-powered product discovery platform, we designed a three-layer architecture:

1. **Knowledge Graph Layer**: Neo4j stores product relationships, categories, and attributes as a graph, enabling complex traversal queries
2. **Vector Search Layer**: Product descriptions and user queries are embedded using sentence transformers and stored in Neo4j vector indices for semantic similarity search
3. **LLM Re-ranking Layer**: Candidate results from vector search are re-ranked using an LLM that considers user context, clickstream signals, and query intent

## Why RAG Over Fine-tuning?

A common question is why use RAG instead of fine-tuning an LLM on your product catalog. The answer comes down to several practical considerations:

- **Freshness**: Product catalogs change constantly. RAG systems can reflect inventory changes instantly without retraining
- **Hallucination Control**: By grounding the LLM's responses in retrieved documents, we significantly reduce the risk of generating fictional product information
- **Cost**: Fine-tuning large models is expensive and needs to be repeated as data changes. RAG leverages pre-trained models with your data at retrieval time
- **Explainability**: With RAG, you can trace exactly which documents influenced a recommendation

## The Role of Knowledge Graphs

One of the key innovations in Nexora was combining vector search with a Neo4j knowledge graph. While vector search excels at finding semantically similar items, a knowledge graph captures structured relationships that embeddings alone might miss:

- Product compatibility relationships
- Brand hierarchies
- Category taxonomies
- User purchase patterns

By querying the graph first to identify related product clusters, then using vector search within those clusters, we achieved much higher precision than either approach alone.

## Clickstream-Driven Re-ranking

The final piece of the puzzle is real-time re-ranking based on user behavior. Using GCP BigQuery to process clickstream data, we build user affinity profiles that inform the LLM re-ranker:

- Products with higher click-through rates for similar user segments get boosted
- Items that frequently lead to purchases from search get prioritized
- Zero-result queries trigger fallback strategies using the LLM's reasoning capabilities

## Results

The combination of these three layers delivered impressive results:

- **52% improvement** in discovery precision
- **38% reduction** in search abandonment
- Sub-200ms latency for the full retrieval + re-ranking pipeline

## Lessons Learned

Building production RAG systems taught me several important lessons:

1. **Chunking strategy matters**: How you split product descriptions into chunks dramatically affects retrieval quality
2. **Hybrid search wins**: Combining keyword search with vector search (using reciprocal rank fusion) consistently outperforms either alone
3. **Evaluation is hard**: Traditional IR metrics like NDCG need to be supplemented with business metrics like conversion rate
4. **Caching is crucial**: LLM calls are expensive — aggressive caching of common query patterns keeps costs manageable
